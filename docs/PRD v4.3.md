## 1. Overview / 系統概覽

### 1.1 Vision

輕量化的年度預算蒐集平台。系統自動派送預填實績費用的 Excel 樣板給各填報單位,單位填寫後上傳至系統。HR 另行匯入各部門人力預算,財務另行匯入各部門公攤費用；兩者獨立管理,不干擾部門 Excel 流程。上階主管與財務透過即時彙整報表 Dashboard 掌握各層填寫狀態,彙整報表（1000處及以上）完整呈現部門預算、人力預算與公攤費用三個維度。系統聚焦於資料的上下傳、版本管理與報表呈現,不涉及線上填報或電子簽核。

| 1000 | 處 | 填報單位 + 彙整報表 Reviewer（含人力/公攤） |
| 0800 | 中心 | 填報單位 + 彙整報表 Reviewer（含人力/公攤） |
| 0500 | BG | 填報單位 + 彙整報表 Reviewer（含人力/公攤） |

**人力預算與公攤費用：** 由 HR 管理員與財務管理員分別匯入,獨立於部門 Excel 之外。彙整報表於 **1000處（含）以上**合併呈現。

## 2. Problem & Goals / 問題與目標

### 2.1 Problem Statement / 問題陳述

目前集團各部門年度預算蒐集仰賴 Excel 透過 Email 往返傳遞,導致版本混亂、財務彙整耗時、無法即時掌握各單位填寫進度。人力預算由 HR 管理,公攤費用由財務計算,兩者目前亦透過 Email 傳遞,缺乏統一管理機制。

### 2.2 Business Goals / 業務目標

- 統一樣板派送,消除版本混亂
- 系統集中接收上傳,取代 Email 附件往返
- HR 人力預算與財務公攤費用統一匯入管理,不依賴 Email
- 即時呈現各層填寫與上傳狀態,取代人工追蹤
- 財務與上階主管可即時查看逐層彙整報表（含人力與公攤）

### 2.3 Non-Goals / 非目標範圍

- 線上填報表單（Excel 離線填寫後上傳）
- 電子簽核流程
- 異常標記子流程
- 多幣別匯率換算（Phase 1）
- ERP / 會計系統 API 整合
- HR 人力預算明細（人數、薪資結構等）,僅需會科層級金額

## 3. User Stories / 用戶故事

### 財務管理員（Finance Admin）

- 身為財務管理員,我需要設定預算週期、維護會科主檔與實際費用資料,讓樣板能正確預填
- 身為財務管理員,我需要系統自動派送 Excel 樣板給所有填報單位,不需逐一寄送
- 身為財務管理員,我需要匯入各部門公攤費用分攤結果,讓彙整報表能完整反映總成本
- 身為財務管理員,我需要即時查看全集團所有單位的上傳狀態,掌握蒐集進度
- 身為財務管理員,我需要查看逐層彙整報表（含人力與公攤）,並可通知特定單位重傳

**驗收條件：**
- 可設定填報年度、截止日期、提醒排程 → FR-001, FR-005
- 可維護會科主檔與實際費用 → FR-007, FR-008
- 週期開放後系統自動發送樣板下載連結（不含 0000公司）→ FR-003, FR-009
- 可上傳公攤費用 CSV/Excel（部門ID、會科代碼、金額）→ FR-027
- 可查看全集團 Dashboard → FR-014
- 可對任一填報單位發送重傳通知 → FR-018, FR-019
- 可匯出彙整報表 → FR-015, FR-017

### HR 管理員（HR Admin）

- 身為 HR 管理員,我需要匯入各部門人力相關會科的預算金額,讓彙整報表能完整反映人力成本

**驗收條件：**
- 可上傳人力預算 CSV/Excel（部門ID、會科代碼、金額）→ FR-024
- 系統驗證部門ID與會科代碼存在於主檔 → FR-024
- 驗證失敗提供列層級錯誤報告 → FR-024
- 同一週期可多次上傳,後次覆蓋前次 → FR-025
- 上傳成功後系統通知財務管理員 → FR-026
- 彙整報表（1000處及以上）即時反映最新人力預算數字 → FR-015, FR-026

### 填報單位主管（Budget Owner）

- 身為填報單位主管,我需要收到系統通知並下載預填實績的 Excel 樣板
- 身為填報單位主管,我需要填妥後上傳至系統,並確認上傳成功

**驗收條件：**
- 收到開放通知含樣板下載連結 → FR-003, FR-010
- 可下載含本單位實績費用與部門代碼的樣板 → FR-009, FR-010
- 上傳後系統驗證格式並顯示確認 → FR-011, FR-013
- 可查看本單位歷史上傳版本 → FR-012
- 收到重傳通知時可重新上傳 → FR-018, FR-011

### 上階主管 Reviewer（1000處 / 0800中心 / 0500 BG / 0000公司）

- 身為上階主管,我需要即時查看轄下各單位的上傳狀態與彙整預算數字（含人力與公攤）
- 身為上階主管,我需要對不符合的單位發送重傳通知

**驗收條件：**
- 可查看管轄範圍內各填報單位的狀態 → FR-014, FR-022
- 可查看彙整報表（部門營運預算 + 人力預算 + 公攤費用三欄並列）→ FR-015
- 可對特定單位發送重傳通知 → FR-018, FR-019
- **0000公司 Reviewer 僅查看全公司彙整報表** → FR-022

### IT 資安稽核人員（IT Security Auditor）

**驗收條件：**
- 可查詢所有操作日誌 → FR-023
- 日誌含使用者、時間、IP、操作內容 → FR-023
- 日誌不可竄改 → FR-023
- 支援多條件篩選 → FR-023

### 4.1 Budget Cycle Management / 預算週期管理

| FR | 功能 | 說明 | 優先級 |
|---|---|---|---|
| FR-001 | 建立預算週期 | 輸入：會計年度、填報截止日、報告幣別。每會計年度僅一個有效週期。 | P0 |
| FR-002 | 確認填報單位清單 | 管理員確認組織樹中 4000部至 0500 BG（含）所有單位的填報清單；6000組、5000課、0000公司不在派送範圍內。**錯誤處理：若組織樹中存在無主管（org_unit_id 無對應 User）之填報單位,系統顯示警告並阻擋週期開放,直到補齊或明確排除。**| P0 |
| FR-003 | 開放週期 | 週期 Draft → Open；系統自動寄送樣板下載通知給所有填報單位主管（不含 0000公司）。**狀態轉換：僅允許 Draft → Open；已 Open 或 Closed 週期不可重複開放。通知發送失敗（Email bounce）於 Dashboard 標示並可重送。** | P0 |
| FR-004 | 監控週期進度 | Dashboard 即時顯示各填報單位狀態：未下載／已下載／已上傳／已通知重傳。**刷新頻率：狀態變更後 ≤ 5 秒內反映；使用者可手動刷新。若無任何填報單位（週期未開放）顯示「尚未開放週期」提示。** | P0 |
| FR-005 | 截止日提醒 | 可設定提醒排程（如截止前 7/3/1 天）自動通知未上傳單位。排程於每日 09:00（伺服器時區）掃描並發送；已上傳單位自動排除。 | P1 |
| FR-006 | 關閉週期 | 管理員手動關閉或截止日到期自動關閉；關閉後不接受新上傳（包含人力預算與公攤費用匯入）。**狀態轉換：Open → Closed 單向；關閉後新上傳請求回傳「週期已關閉」錯誤訊息並記錄於稽核日誌。管理員可於設定時限內（例：關閉後 7 天）重開週期,由系統管理員核准並留存原因。** | P0 |

## 6. Workflow / 作業流程

1. **【系統設定】** 管理員維護組織樹、標記填報單位（4000部至 0500 BG）、設定會科主檔（含人力類別與公攤類別標記）、匯入當年度實績費用
2. **【週期開放】** 財務設定年度與截止日,開放週期；系統自動為每個填報單位（不含 0000公司）產生個人化 Excel 樣板,Email 通知含下載連結
3. **【下載樣板】** 填報單位主管收到通知後,下載含本單位實績的 Excel 樣板（不含人力/公攤欄位）
4. **【離線填寫】** 主管依營運會科填寫預算金額（離線,不在系統內）
5. **【上傳預算】** 填妥後登入系統上傳；系統驗證格式,成功後建立版本紀錄,通知直屬上階主管
6. **【HR 匯入人力預算】** HR 管理員於系統上傳人力預算 CSV/Excel（部門ID、會科代碼、金額）；驗證通過後系統建立版本紀錄,通知財務管理員
7. **【財務匯入公攤費用】** 財務管理員於系統上傳公攤費用 CSV/Excel（部門ID、會科代碼、金額）；驗證通過後系統建立版本紀錄
8. **【即時彙整】** 上階主管與財務隨時登入查看管轄範圍 Dashboard（上傳狀態）與彙整報表；**1000處（含）以上的彙整報表並列顯示：部門營運預算 + 人力預算 + 公攤費用**；0000公司 Reviewer 僅查看全公司彙整報表
9. **【通知重傳】** 財務或上階主管發現數字不符合時,點擊「通知重傳」並填寫說明；系統 Email 該單位主管,含說明與樣板下載連結；Dashboard 標記「已通知重傳」
10. **【重傳】** 填報單位修改後重新上傳；系統保留舊版本,新版本成為有效版本,自動更新上層彙整報表
11. **【關閉週期】** 截止日到期或管理員手動關閉；關閉後不接受新上傳及新匯入；財務匯出正式彙整報表

## 7. Security Specifications / 資安規範

本節以「能力」（capability）描述,具體實作技術（協定、演算法、版本）統一定義於 §14 Technology Constraints,以利日後替換而不影響需求文件。

| 項目 | 能力規範 |
|---|---|
| 身分驗證 | 強制透過企業既有身分目錄進行 federated SSO；禁止本地帳號；支援多因子驗證 |
| 傳輸安全 | 全站僅允許加密傳輸；API 呼叫需附有效短期憑證（token-based） |
| Session 管理 | 閒置 30 分鐘自動登出；憑證具有效期限控制與刷新機制 |
| 網路存取控制 | 僅限企業內網或 VPN；支援 IP 白名單 |
| 資料保護 | 上傳檔案靜態加密儲存；資料庫敏感欄位欄位級加密；禁止應用程式以外管道直接存取生產資料庫 |
| 稽核日誌 | append-only,具完整性保護機制（防竄改）；保留至少 5 年 |
| 災難復原 | 每日自動備份；RPO ≤ 24 小時；RTO ≤ 4 小時 |

## 14. Technology Constraints / 技術限制

§7 Security 與 §FR-021 等需求以「能力」描述,本節定義對應的技術實作。實作可在不更動需求文件的前提下被替換。

### Decided

| 項目 | 技術選擇 | 對應需求 |
|---|---|---|
| 身分驗證協定 | OIDC 或 SAML 2.0（依企業 IdP 支援擇一） | FR-021, §7 身分驗證 |
| API 憑證 | JWT（短期, 含 refresh） | §7 傳輸安全, Session 管理 |
| 傳輸加密 | HTTPS / TLS 1.2 以上 | §7 傳輸安全, NFR-SEC-001 |
| 敏感欄位加密 | AES-256（資料庫欄位級） | §7 資料保護 |
| 稽核日誌完整性 | append-only + 雜湊鏈 | §7 稽核日誌, FR-023 |
| 前端 | React Web App | §1.3 |
| 資料庫類型 | 關聯式資料庫 | §10 |
| ARIA 語意關聯 | `aria-describedby` 將錯誤訊息連結至輸入欄位 | §8.4 |

- 後端語言／框架
- 資料庫引擎（PostgreSQL、SQL Server 等）
- 部署策略
- Excel 產生／解析 library
